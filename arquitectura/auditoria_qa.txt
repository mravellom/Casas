#1 — Race Condition en Upsert de Propiedades
   Archivo:   app/workers/scrape_job.py (líneas 228-243)
   Severidad: CRÍTICO

   Problema: El patrón check-then-insert tiene race condition entre
   el SELECT y el INSERT. Dos jobs concurrentes podrían crear
   duplicados a pesar del unique constraint.

   Código afectado:
     existing = result.scalar_one_or_none()
     if existing:
         # update...
     else:
         # insert...  ← Race condition aquí

   Impacto: Duplicados en BD; violación de integridad de datos.

   Fix: Usar INSERT ... ON CONFLICT DO UPDATE de PostgreSQL
   o locking explícito con SELECT ... FOR UPDATE.


#2 — Status Code Incorrecto en API de Propiedades
   Archivo:   app/api/properties.py (línea 83)
   Severidad: CRÍTICO

   Problema: Cuando no se encuentra una propiedad, retorna un dict
   con error pero SIN status code HTTP correcto:

     if not prop:
         return {"error": "Propiedad no encontrada"}, 404

   FastAPI ignora la tupla — siempre devuelve 200.

   Impacto: Frontend no puede detectar errores; HTTP status ignorado.

   Fix: Usar raise HTTPException(status_code=404, detail="...")


#3 — Timeout en API de UF sin Circuit Breaker
   Archivo:   app/analysis/uf_converter.py (líneas 26-48)
   Severidad: CRÍTICO

   Problema: La llamada a mindicador.cl tiene timeout de 10s pero
   si la API falla repetidamente, no hay circuit breaker. El pipeline
   entero puede quedar colgado.

   Impacto: Pipeline se detiene; no se envían alertas.

   Fix: Implementar circuit breaker (tenacity con max_retries)
   y cachear valor UF en BD como fallback.


#4 — Validación Faltante en Parámetro commune
   Archivo:   app/api/opportunities.py (líneas 31-37)
   Severidad: CRÍTICO

   Problema: El parámetro commune se usa directamente en la query
   sin validar que sea una comuna válida del sistema. SQLAlchemy
   protege contra SQL injection, pero permite filtrar por comunas
   inexistentes sin feedback.

   Impacto: Posible información no deseada; sin feedback al usuario.

   Fix: Validar commune contra settings.target_communes.

ISSUES ALTOS (8)
================================================================================

#5 — Credenciales Hardcodeadas en Config
   Archivo:   app/config.py (líneas 6, 9)
   Severidad: ALTO

   Problema: Credenciales por defecto en el código:
     database_url: str = "postgresql+asyncpg://inmoalert:inmoalert_pass@..."
     redis_url: str = "redis://redis:6379/0"

   Impacto: Credenciales expuestas en control de versiones.
   Fix: Remover defaults; requerir env vars obligatorias.


#6 — CORS Hardcodeado para Localhost
   Archivo:   app/main.py (líneas 65-70)
   Severidad: ALTO

   Problema: CORS permite all methods y headers solo desde localhost:3000.
   En producción esto no funcionará.

     allow_origins=["http://localhost:3000"],
     allow_methods=["*"],
     allow_headers=["*"],

   Fix: Cargar origins desde variable de entorno; restringir methods.


#7 — Sin Autenticación en Endpoints Admin
   Archivo:   app/api/health.py
   Severidad: ALTO

   Problema: TODOS los endpoints admin están sin autenticación:
   - POST /admin/scrape/trigger (cualquiera puede disparar scraping)
   - GET /admin/metrics (datos internos expuestos)
   - GET /admin/feedback/stats (datos de negocio expuestos)

   Impacto: DoS, ejecución no autorizada, exposición de datos.
   Fix: Agregar autenticación (JWT, API key) con roles.


#8 — Fallo Silencioso del Bot Telegram
   Archivo:   app/main.py (líneas 42-44)
   Severidad: ALTO

   Problema: Si el bot falla al iniciar, solo logea warning y continúa.
   Las alertas nunca se enviarán pero la app parece healthy.

   Impacto: Pérdida silenciosa de alertas.
   Fix: Verificar estado del bot antes de enviar; alertar en health check.


#9 — Error Genérico en Health Check de BD
   Archivo:   app/api/health.py (líneas 29-37)
   Severidad: ALTO

   Problema: except Exception swallow ALL errors:
     except Exception:
         db_status = "error"  # ¿Qué error? Imposible debuggear.

   Fix: Loggear la excepción; diferenciar errores esperados vs bugs.


#10 — Queries N+1 en API de Oportunidades
    Archivo:   app/api/opportunities.py (líneas 49-90)
    Severidad: ALTO

    Problema: Carga TODOS los market averages en memoria para cada
    request, luego busca en dict por cada propiedad.

    Impacto: Memory bloat con datasets grandes.
    Fix: Usar JOIN en la query SQL.


#11 — Sin Límite de Páginas en Paginación
    Archivo:   app/api/properties.py (líneas 14-21)
    Severidad: ALTO

    Problema: Se puede pedir page=999999 con limit=100. No hay
    cursor-based pagination.

    Impacto: DoS por offset gigantes; slow DB scans.
    Fix: Implementar cursor-based pagination o limitar offset.


#12 — Deduplicación No Transaccional
    Archivo:   app/analysis/filters.py (líneas 176-199)
    Severidad: ALTO

    Problema: Carga todas las propiedades, compara en Python,
    luego actualiza. Otro proceso puede modificar datos entre medio.

    Fix: Usar deduplicación a nivel de BD o locking.
